## Closed contract

- Reads are limited to the required order, order items, entries, and selected local/event source.
- Order, item, entry, source, paid-state, count, identity, and deliverability data are validated before message construction.
- Local and event source lookup are mutually selected from the order contract.
- Missing source names may use a stable fallback; malformed source rows cannot hide behind it.
- Shuffled database rows produce the same canonical data and B3B1 hash.

## Introduced invariants

- The adapter is read-only and uses literal projections/filters.
- Aborts are checked at every read boundary and immediately before success.
- Abort takes precedence over transport or validation errors when already authoritative.
- Error results expose stable sanitized codes, never raw transport details, buyer PII, or tokens.
- Successful returned data is defensive and deeply frozen.

## Principal validations

- Paid/source/item/entry/count/deliverability, canonical ordering, abort, immutability, and sanitization cases in `accessEmailMessageData.test.ts`.
- Exact Supabase table/projection/filter/signal behavior and malformed/error cases in `accessEmailMessageDataSupabase.test.ts`.
- Integration remains provider-free and does not call remote services in tests.

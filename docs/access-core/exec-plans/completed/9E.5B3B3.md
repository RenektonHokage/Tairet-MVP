## Closed contract

- Provider input requires idempotency key, request-payload hash, template version, and canonical message.
- Provider outcomes are limited to `accepted`, `failed_retryable`, `failed_terminal`, and `ambiguous`.
- Resend uses the durable idempotency key and the exact composed abort signal.
- Only a well-formed nonempty provider message ID is conclusive acceptance.
- Rate limits may supply bounded retry guidance; timeouts, transport uncertainty, contradictory shapes, and unknown failures remain ambiguous or conservative.

## Introduced invariants

- Provider message text is never outcome authority.
- Invalid input fails before client construction, scheduling, or SDK use.
- A timeout/external abort after provider start is ambiguous.
- Late provider resolution or rejection after a returned timeout is inert and safely consumed.
- Returned outcomes contain only frozen allowlisted fields and no provider payload/secret leakage.
